# 复杂任务分解模板 (Complex Decomposition Template)

用于将大型、多步骤的任务分解成可管理的阶段和步骤。

## Template Structure

```
## 最终目标 (Final Objective)
[Clear, specific, measurable outcome]

## 阶段概述 (Phase Overview)
[High-level breakdown of major phases]

## 详细步骤 (Detailed Steps)
### Phase 1: [Phase Name]
- Step 1.1: [Detailed action]
- Step 1.2: [Detailed action]
- Step 1.3: [Detailed action]

### Phase 2: [Phase Name]
- Step 2.1: [Detailed action]
- Step 2.2: [Detailed action]

## 成功标准 (Success Criteria)
[How to validate each phase]

## 关键注意事项 (Key Considerations)
[Important rules, constraints, edge cases]

## 备选方案 (Fallback Options)
[Alternative approaches if something fails]

## 输出格式 (Output Format)
[How to present the results]
```

## Complete Example: Data Analysis Report Generation

```
## 最终目标
生成一份全面的、可执行的分析报告，包含数据洞察、关键发现和明确的建议。报告应该能在15分钟内让高管理解核心信息。

## 阶段概述
1. **数据理解阶段** - 探索和清洁数据
2. **分析阶段** - 提取洞察和发现模式
3. **结论阶段** - 生成建议和行动项
4. **报告生成阶段** - 组织成专业文档

## 详细步骤

### Phase 1: 数据理解 (Data Understanding)
- **Step 1.1**: 检查数据完整性
  - 验证所有预期的列都存在
  - 检查缺失值（>5%则标记为警告）
  - 确认数据类型正确
  
- **Step 1.2**: 数据清洁
  - 移除重复记录
  - 处理异常值（记录处理决策）
  - 标准化格式（日期、货币、分类）
  
- **Step 1.3**: 统计摘要
  - 计算基本统计量(均值、中位数、标准差)
  - 识别数据分布
  - 记录任何异常观察

**验证检查清单:**
- [ ] 数据行数和列数与预期一致
- [ ] 没有无法解析的值
- [ ] 日期范围合理
- [ ] 数值范围在预期内

### Phase 2: 深度分析 (In-Depth Analysis)
- **Step 2.1**: 趋势分析
  - 按时间段聚合数据
  - 计算同比/环比增长
  - 识别上升/下降周期
  
- **Step 2.2**: 分类分析
  - 按关键维度(地区、产品、客户等)分组
  - 计算各分组的关键指标
  - 识别表现最好/最差的分组
  
- **Step 2.3**: 相关性分析
  - 找出主要变量之间的关系
  - 识别驱动因素
  - 评估关系强度

**验证检查清单:**
- [ ] 所有分析都基于清洁的数据
- [ ] 计算方法已记录
- [ ] 发现能用原始数据验证
- [ ] 结论基于统计证据

### Phase 3: 结论和建议 (Conclusions & Recommendations)
- **Step 3.1**: 综合关键发现
  - 优先排列3-5个核心发现（按影响力）
  - 每个发现用数据支持
  - 解释"为什么这很重要"
  
- **Step 3.2**: 生成可执行的建议
  - 针对每个发现提出2-3个行动项
  - 估算预期影响（高/中/低）
  - 概述实施难度
  
- **Step 3.3**: 识别风险和限制
  - 数据质量限制
  - 分析假设
  - 可能的偏差

**验证检查清单:**
- [ ] 每个建议都可追溯到数据发现
- [ ] 建议是具体的、可衡量的
- [ ] 难度和影响评估是现实的
- [ ] 风险被充分讨论

### Phase 4: 报告生成 (Report Generation)
- **Step 4.1**: 组织内容
  - 执行摘要（1-2段）
  - 高级发现（3-5项）
  - 详细分析（支持发现）
  - 建议（优先排列）
  - 附录（方法论、图表）
  
- **Step 4.2**: 可视化
  - 为关键指标创建图表
  - 使用表格显示比较
  - 用颜色突出显示趋势
  
- **Step 4.3**: 格式和审查
  - 一致的标题和格式
  - 清晰的图表标签和标题
  - 检查拼写和语法

**验证检查清单:**
- [ ] 报告长度适当（不超过10页）
- [ ] 执行摘要独立成立
- [ ] 所有数字和引用都准确
- [ ] 格式专业且易阅读

## 成功标准

**整体报告:**
- ✅ 非技术性高管在15分钟内理解核心信息
- ✅ 所有关键数据点都有来源和方法论说明
- ✅ 建议是可操作的，并有清晰的步骤

**数据部分:**
- ✅ 数据质量问题被识别和记录
- ✅ 统计假设被陈述
- ✅ 计算可被复现

**建议部分:**
- ✅ 每个建议都基于数据证据
- ✅ 优先级清晰
- ✅ 实施路径明确

## 关键注意事项

### 数据质量 (Data Quality)
- 如果缺失值超过30%的字段，考虑排除
- 如果检测到异常值占>5%，需要调查
- 始终记录数据清洁决策

### 分析深度 (Analysis Depth)
- 不要过度分析微不足道的差异（<2%）
- 关注能驱动业务决策的洞察
- 避免数据挖掘陷阱（虚假相关性）

### 受众意识 (Audience Awareness)
- 避免技术术语，用业务语言说话
- 每个发现都要回答"这对我们意味着什么？"
- 提供背景信息帮助理解结果

## 备选方案 (Fallback Options)

### 如果数据不完整
```
→ 记录缺失数据
→ 进行敏感性分析（最佳/最坏情况）
→ 在报告中清楚地标注：基于不完整数据的发现
```

### 如果发现不显著
```
→ 重点放在"没有发现显著变化"本身
→ 调查为什么预期的某些趋势没有出现
→ 建议监控以在未来检测变化
```

### 如果有相互矛盾的指标
```
→ 深入调查根本原因
→ 解释不一致之处
→ 优先考虑更可靠的数据源
```

## 输出格式

使用Markdown格式：
```markdown
# [报告标题]

## 执行摘要
[1-2段，回答：现在的情况？为什么重要？建议是什么？]

## 关键发现
1. **发现 1**: [标题]
   - 支持数据点
   - 业务影响
   
2. **发现 2**: [标题]
   - 支持数据点
   - 业务影响

## 详细分析
[按发现组织]

## 建议
1. **行动项 1**: [具体步骤] - 难度: [高/中/低], 预期影响: [高/中/低]
2. **行动项 2**: [具体步骤] - 难度: [高/中/低], 预期影响: [高/中/低]

## 方法论和限制
[数据质量 | 分析方法 | 已做假设 | 已知限制]
```

## 适应指南

| 场景 | 要调整的步骤 |
|-----|-------------|
| **营销活动分析** | 第2.2步中添加细分 (年龄、地理位置等) |
| **产品性能评估** | 第2.3步中添加竞争对比 |
| **运营效率审查** | Phase 1中添加流程映射 |
| **财务审计** | 第3步中添加合规性检查 |

---

## 关键指标示例

根据您的域名调整分析关键指标：

- **营销**: CTR, 转化率, CAC, ROAS, 保留率
- **产品**: DAU, NPS, 功能采用率, 流失率
- **运营**: 成本效率, 周期时间, 错误率, 产能利用率
- **财务**: 利润率, 应收账款周期, 现金流, ROI
