# 常见提示词模式 (Common Patterns)

经过验证的、可重复使用的提示词结构和框架。

## 1. 思维链模式 (Chain-of-Thought Pattern)

**用途**: 复杂推理，需要逐步思考

**模式**:
```
任务: [你的任务]

按照以下步骤思考:
1. [第一步结果应该是什么]
2. [第二步结果应该是什么]
3. [第三步结果应该是什么]
...

最终回答: [总结前面步骤得出的结论]

示例:
[展示思维过程的示例]
```

**示例应用**:
```
任务: 确定这是否是一项合理的软件架构

按照以下步骤思考:
1. 识别所建议的架构设计的关键组件
2. 评估每个组件的适当性
3. 检查组件之间的关系
4. 评估可扩展性和维护性
5. 考虑可能的故障点

最终回答: 是否合理，以及为什么
```

## 2. 角色扮演模式 (Role Playing Pattern)

**用途**: 当AI应该采用特定的视角或角色时

**模式**:
```
你是: [清晰的角色定义]
你的背景: [1-2句话的经验]
你的风格: [如何交流]

当被问及 [话题] 时，从 [角色] 的视角回答
```

**示例应用**:
```
你是一个充满热情的创业者顾问，有20年的初创公司经验。
你见过什么能成功，什么能失败。
你直率、实用、鼓励创业者考虑现实。

当被问及一个商业想法时，从顾问的视角：
1. 评估市场潜力
2. 确定关键风险
3. 建议思考什么
```

## 3. 情景分析模式 (Scenario Analysis Pattern)

**用途**: 探索多个可能结果或方法

**模式**:
```
对于 [情景/问题]，考虑:

情景A: [一种方法]
- 优点: ...
- 缺点: ...
- 最好情况: ...
- 最坏情况: ...

情景B: [另一种方法]
- 优点: ...
- 缺点: ...
- 最好情况: ...
- 最坏情况: ...

建议: [推荐的方法和原因]
```

**示例应用**:
```
对于更新我们的支付系统，考虑:

方案A: 优雅迁移（逐步替换）
- 优点: 低风险、最小停机
- 缺点: 需要维护两个系统
- 最好情况: 无缝过渡
- 最坏情况: 数据同步问题

方案B: 直接替换（一次性切换）
- 优点: 快速、完全替换
- 缺点: 高风险、可能停机
- 最好情况: 快速、干净的转换
- 最坏情况: 数据丢失，系统宕机
```

## 4. 第一性原理模式 (First Principles Pattern)

**用途**: 从基础原理出发解决问题

**模式**:
```
问题: [你的实际问题]

第一性原理分析:
1. 基本假设: [什么是真的]
2. 分解问题: [主要组成部分]
3. 从基础重建: [从零开始思考]
4. 重新组合: [最优方式]

建议: [基于第一性原理的建议]
```

**示例应用**:
```
问题: 如何改进新员工入职

第一性原理分析:
1. 根本需求: 新员工需要信息、关系、表现期望
2. 核心障碍: 信息过载、孤立、不清楚优先级
3. 关键杠杆: 清晰的沟通、伙伴分配、可执行的目标
4. 最优流程: 系统化的信息、专业配对、频繁的反馈

建议: 开发清单→分配伙伴→设置周里程碑
```

## 5. 对比分析模式 (Comparative Analysis Pattern)

**用途**: 比较选项、产品或方法

**模式**:
```
比较: [选项A] vs [选项B] vs [选项C]

在以下维度上:
1. [维度1]: A的...  B的...  C的...
2. [维度2]: A的...  B的...  C的...
3. [维度3]: A的...  B的...  C的...

总结:
- [选项A]: 最适合...
- [选项B]: 最适合...
- [选项C]: 最适合...

推荐: [选项] 用于 [你的用例]，因为...
```

**示例应用**:
```
比较: React vs Vue vs Angular

在以下维度上:
1. 学习曲线: React简陡，Vue中等，Angular陡
2. 灵活性: React最高，Vue中等，Angular严格
3. 社区和生态: React最大，Vue中等，Angular企业
4. 性能: React快，Vue快，Angular中等

推荐: 新项目选React（大社区），小项目选Vue（简单）
```

## 6. "为什么"链接模式 (Why Chaining Pattern)

**用途**: 深掘问题的根本原因

**模式**:
```
问题: [表面问题]

为什么? [第一层原因]
为什么? [第二层原因]
为什么? [第三层原因]
为什么? [第四层原因]
为什么? [根本原因]

根本解决方案: [解决根本原因]
```

**示例应用**:
```
问题: 用户不使用新功能

为什么? 他们不知道它存在
为什么? 我们没有有效沟通
为什么? 营销计划不周
为什么? 我们没有针对早期采用者的策略
根本原因: 缺乏后期营销流程

解决方案: 建立产品发布和营销清单
```

## 7. 假设检验模式 (Assumption Testing Pattern)

**用途**: 识别和验证关键假设

**模式**:
```
计划/想法: [你的计划]

关键假设:
1. 我们假设 [假设1] 是真的
2. 我们假设 [假设2] 是真的
3. 我们假设 [假设3] 是真的

验证这些的方法:
1. [如何测试假设1] → 预期结果: ...
2. [如何测试假设2] → 预期结果: ...
3. [如何测试假设3] → 预期结果: ...

风险: 如果这些假设错了会发生什么
```

**示例应用**:
```
计划: 推出按年订阅选项

关键假设:
1. 年度用户存在足够大的受众
2. 年度价格点会有吸引力
3. 它不会伤害月度收入

验证方法:
1. 调查现有用户→看有多少对年度感兴趣
2. A/B测试价格点→看转化率
3. 建模收入影响→看净利益

如果错了会发生什么:
- 假设1错: 推出将被冷漠对待
- 假设2错: 没人会转换
- 假设3错: 收入下降而损失收益
```

## 8. SCQA框架 (Situation-Complication-Question-Answer)

**用途**: 清晰地陈述问题和所需要的帮助

**模式**:
```
情况(Situation): [背景 - 什么是真的]
复杂化(Complication): [问题 - 什么出错了]
问题(Question): [具体问题 - 你需要什么帮助]
预期答案(Answer): [你期望的帮助类型]

额外背景: [任何其他相关信息]
```

**示例应用**:
```
情况: 我们是一个30人的SaaS公司，有稳定的MRR
复杂: 我们的客户流失率从2%上升到5%
问题: 我们应该采取什么步骤来诊断和解决流失率
答案: 我需要分析框架和可采取的行动列表

额外背景: 最近我们更改了定价模型
```

## 9. CEO邮件模式 (CEO Email Pattern)

**用途**: 清晰向管理层传达

**模式**:
```
主题: [清晰的一行摘要]

关键点(一句话):
[一句话总结你需要知道的一切]

背景:
[1-2段说明情况]

建议:
[清晰的建议或选项]

时间线:
[什么时候需要决策]

所有权:
[谁负责]
```

**示例应用**:
```
主题: 第二季度营销预算提案

关键点: 建议将营销预算增加25%投资增长，预期ROI 3:1

背景:
[为什么现在是时候了，市场机会等]

建议:
选项A: 增加25%(推荐)
选项B: 增加10%(保守)
选项C: 保持不变(风险)

时间线: 需要在周五前批准
所有权: 营销副总裁
```

## 10. 递归细化模式 (Recursive Refinement Pattern)

**用途**: 逐步改进或精炼想法

**模式**:
```
初步想法: [你的初步想法]

第一次细化:
[改进版本1] - 改进: ...

第二次细化:
[改进版本2] - 进一步改进: ...

第三次细化:
[改进版本3] - 最终改进: ...

最终输出:
[精炼的想法]
```

**示例应用**:
```
初步想法: 更好的入职流程

第一次细化:
- 添加视频教程
- 创建快速入门指南
改进: 现在有文档，但还没有个人化

第二次细化:
- 添加欢迎电话
- 个性化电子邮件系列
- 分配伙伴
改进: 现在个性化，但可能过多

第三次细化:
- 清单，不是电子邮件
- 伙伴检查在关键点
- 灵活的时间表
最终: 系统化和人为的混合
```

## 11. 红队打击(Red Team)模式

**用途**: 为想法找失败的论据或风险

**模式**:
```
提案: [你的想法]

红队论据 - 为什么这会失败:
1. 论据A: ...  如何回应: ...
2. 论据B: ...  如何回应: ...
3. 论据C: ...  如何回应: ...

最强的论据: [最有说服力的反对意见]

如果对方是对的会怎样: [最坏的情况]
```

**示例应用**:
```
提案: 提高定价20%

红队论据:
1. 客户将离开 → 但我们的NPS很高
2. 合并将减少 → 但成本上升，需要覆盖
3. 竞争对手不会提价 → 但我们有独特价值

最强论据: 离客户流失可能抵消额外容量

风险缓解: 分阶段推出，有保留对话
```

## 12. 反向工程模式 (Reverse Engineering Pattern)

**用途**: 向后工作从你想要的结果

**模式**:
```
最终目标: [你想要的结果]

第4步: 什么必须为真 → 目标

第3步: 什么必须为真 → 第4步

第2步: 什么必须为真 → 第3步

第1步: 从这里开始 → 第2步

行动计划: [步骤1,2,3,4]
```

**示例应用**:
```
最终目标: 赢得企业客户

第4步: 他们必须购买 ← 需要信任、ROI证明、参考资料
第3步: 他们必须想要 ← 需要产品演示、案例研究、评估
第2步: 他们必须知道 ← 需要销售对话、公司研究、
第1步: 他们必须关注 ← 需要营销活动、出站、活动

行动: 目标出站 → 获胜→ 产品演示 → 签署
```

## 组合模式: 完整示例

有时候，为了最佳结果，你会组合**多个模式**:

```
[开业SCQA] 
情况: 我们更新了一个功能但采用缓慢

[应用思维链]
第1步: 分析采用指标
第2步: 确定采用障碍
第3步: 评估解决方案

[应用情景分析]
情景A: 供教育(更好的文档，演练)
情景B: 激励(显示ROI)
情景C: 强制(鼓励迁移)

[应用红队]
为什么这会失败? 用户觉得被强制 → 如何回应?

[应用反向工程]
最终: 采用率>80%
← 需要什么?
```
