💡 什么是 Agent Skills？

Agent Skills 是模块化、自包含的知识包，通过提供专业知识、工作流程和工具集成来扩展 Claude 的能力。

💭 可以把它们想象成 AI 的"专业培训课程"

Agent Skills 是专门设计的知识模块，它们像是给 AI 提供的专业培训材料，让通用的 AI 变成某个领域的专家。

🎯 核心价值

🧠 从通用到专业

将 Claude 从通用代理转变为配备特定领域程序性知识的专业代理

📦 模块化设计

自包含、可复用、易于维护和分享的知识单元

⚡ 高效上下文管理

渐进式加载，只在需要时载入相关内容

🔧 工具整合

包含脚本、文档和资源，实现端到端的任务处理

Skills 提供四大核心价值：专业化转换、模块化设计、高效的上下文管理，以及完整的工具整合能力。

🎁 Skill 提供什么？

1️⃣ 专业化工作流

  • 特定领域的多步骤流程
  • 例如：PDF 处理、数据分析、代码审查
  • 提供清晰的步骤指引和决策逻辑
第一类是专业化工作流，它为特定任务提供详细的操作步骤和决策指引。

2️⃣ 工具集成

  • 与特定文件格式或 API 协同工作
  • 例如：Excel 操作、数据库查询、Web API 调用
  • 内置脚本和工具函数
第二类是工具集成，提供与外部系统和文件格式交互的能力。

3️⃣ 领域专业知识

  • 公司特定的知识、模式、业务逻辑
  • 例如：内部 API 文档、数据库 Schema、编码规范
  • AI 模型无法预训练的专有信息
第三类是领域专业知识，这些是 AI 模型训练时无法获得的公司或领域特有的信息。

4️⃣ 捆绑资源

  • 脚本：可执行的 Python、Bash 等代码
  • 参考文档：详细的技术文档和示例
  • 资源文件：模板、图标、字体等
第四类是捆绑资源，包括可执行脚本、参考文档和各种资源文件。

🏗️ Skill 结构解析

📁 标准目录结构


skill-name/
├── SKILL.md          # 核心文档（必需）
│   ├── YAML 元数据
│   └── Markdown 指令
└── 捆绑资源（可选）
    ├── scripts/      # 可执行脚本
    ├── references/   # 参考文档
    └── assets/       # 输出资源
每个 Skill 都遵循标准的目录结构，核心是 SKILL.md 文件，可选的捆绑资源按用途分类。

📄 SKILL.md 组成

YAML 元数据


---
name: skill-name
description: 何时使用此技能
license: LICENSE.txt
---

触发条件和描述

Markdown 正文

  • 使用说明
  • 工作流程
  • 示例代码
  • 最佳实践

技能触发后载入

SKILL.md 由两部分组成：YAML 元数据决定何时触发技能，Markdown 正文包含详细的使用说明。

📦 捆绑资源详解

📜 scripts/ - 可执行脚本

用途：需要确定性可靠性或重复编写的任务

示例：rotate_pdf.py - PDF 旋转脚本

优势：Token 高效、确定性、可直接执行

📚 references/ - 参考文档

用途：需要时载入上下文的文档材料

示例：api_docs.md, finance.md

优势：保持 SKILL.md 简洁，按需加载

🎨 assets/ - 输出资源

用途：输出中使用的文件（不载入上下文）

示例：logo.png, template.html

优势：分离输出资源与文档

三类捆绑资源各有用途：scripts 用于确定性任务，references 用于参考文档，assets 用于输出资源。

⚖️ 核心设计原则

原则 1: 简洁至上

上下文窗口是公共资源 🌐

  • 默认假设：Claude 已经很聪明
  • 只添加 Claude 没有的上下文
  • 每段信息都要证明其 Token 成本的合理性
  • 优先使用简洁示例而非冗长解释
第一原则是简洁至上。上下文窗口是共享资源，要珍惜使用，只添加真正必要的信息。

原则 2: 设置适当的自由度

🌊 高自由度（文本指令）

多种方法都有效，决策依赖上下文

🛤️ 中等自由度（伪代码 + 参数）

存在首选模式，允许一些变化

🎯 低自由度（具体脚本）

操作脆弱易错，一致性至关重要

💡 就像探索路径：悬崖峭壁需要护栏（低自由度），开阔平原允许多条路线（高自由度）

第二原则是设置适当的自由度。根据任务的脆弱性和可变性，选择合适的指令形式。

原则 3: 渐进式披露

三层加载系统 🔄

层级 1：元数据（name + description）

始终在上下文中 (~100 词)

层级 2：SKILL.md 正文

技能触发时加载 (<5k 词)

层级 3：捆绑资源

Claude 按需读取（不限）

第三原则是渐进式披露，通过三层加载系统高效管理上下文，只在需要时加载相关内容。

🛠️ 实战模式

渐进式披露的具体应用

模式 1: 高层指南 + 引用


# PDF 处理

## 快速开始
使用 pdfplumber 提取文本：
```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

## 高级功能
- **表单填充**: 参见 [FORMS.md](FORMS.md)
- **API 参考**: 参见 [REFERENCE.md](REFERENCE.md)
- **示例集合**: 参见 [EXAMPLES.md](EXAMPLES.md)

✅ Claude 仅在需要时读取 FORMS.md、REFERENCE.md 或 EXAMPLES.md

模式一：在主文档中提供概览和快速开始，将详细内容拆分到独立文件，按需引用。

模式 2: 领域特定组织


bigquery-skill/
├── SKILL.md              # 概览和导航
└── reference/
    ├── finance.md        # 财务、计费指标
    ├── sales.md          # 销售机会、管道
    ├── product.md        # API 使用、功能
    └── marketing.md      # 营销活动、归因

✅ 用户询问销售指标时，Claude 仅读取 sales.md

模式二：按领域或功能组织内容，避免加载无关上下文，提高效率。

模式 3: 条件式详情


# DOCX 处理

## 创建文档
使用 docx-js 创建新文档。参见 [DOCX-JS.md](DOCX-JS.md)

## 编辑文档
对于简单编辑，直接修改 XML。

**对于修订跟踪**: 参见 [REDLINING.md](REDLINING.md)
**对于 OOXML 详情**: 参见 [OOXML.md](OOXML.md)

✅ 仅当用户需要这些功能时，Claude 才读取相应文档

模式三：展示基础内容，将高级功能链接到专门文档，实现条件加载。

⚠️ 重要指南

❌ 避免深度嵌套引用

保持引用层级为一层，所有引用文件应直接从 SKILL.md 链接

✅ 构建长引用文件

超过 100 行的文件应在顶部包含目录，便于 Claude 预览全貌

实战中要注意两点：避免过深的引用嵌套，为长文件提供目录导航。

🚀 创建你的第一个 Skill

步骤 1: 明确目标

问自己这些问题：

  • 🎯 这个 Skill 要解决什么问题？
  • 👥 谁会使用它？在什么场景下？
  • 📊 Claude 缺少哪些关键信息？
  • 🔧 需要什么工具或脚本？
  • 📝 如何判断任务是否成功完成？
创建技能的第一步是明确目标，回答这五个关键问题。

步骤 2: 设计结构


# 创建基本结构
my-skill/
├── SKILL.md           # 从这里开始
└── (根据需要添加)
    ├── scripts/
    ├── references/
    └── assets/

💡 提示：先完成 SKILL.md，再考虑是否需要其他资源

第二步是设计结构，从 SKILL.md 开始，根据实际需要添加其他资源。

步骤 3: 编写 SKILL.md


---
name: my-awesome-skill
description: 简洁描述何时使用此技能，以及它能做什么
license: MIT
---

# My Awesome Skill

## 概述
简要说明这个技能的用途和价值

## 使用场景
列出适用的具体场景

## 工作流程
1. 第一步做什么
2. 第二步做什么
3. ...

## 示例
提供清晰的代码示例

## 注意事项
列出重要的注意事项或限制
第三步是编写 SKILL.md，遵循标准格式，包含元数据和结构化的正文内容。

步骤 4: 迭代优化

  • 测试：实际使用，看是否达到预期
  • 简化：删除冗余内容，保持简洁
  • 拆分：超过 500 行？考虑拆分到引用文件
  • 优化：调整自由度，确保适合任务类型

🎯 记住：好的 Skill 是迭代出来的！

最后一步是迭代优化，通过实际使用不断改进技能的质量和效率。

✅ 最佳实践清单

✅ 应该做的

✅ 保持 description 清晰、全面（这是触发条件）

✅ 使用简洁示例替代冗长解释

✅ 根据任务脆弱性调整自由度

✅ 将详细内容拆分到引用文件

✅ 为长文件提供目录导航

这些是创建技能时应该遵循的最佳实践。

❌ 不应该做的

❌ 不要创建 README.md 等辅助文档

❌ 不要假设 Claude 不知道基础知识

❌ 不要在 SKILL.md 和 references 中重复信息

❌ 不要创建深度嵌套的引用结构

❌ 不要让 SKILL.md 超过 500 行（考虑拆分）

同时也要避免这些常见的错误做法。
